# Investor Assistant Design

**Date:** 2026-03-24

## Goal

将 Project Sentinel 从“会发日报的投研工具”重构为“以当前持仓和少量观察名单为中心的中长期投资助手”。

重构后的系统不追求高频预测，也不做全市场选股机，而是稳定回答 3 个问题：

- 现在该不该继续拿
- 该把钱加到哪里
- 在什么条件下必须撤退

## Product Positioning

系统面向单一投资者账户，优先服务真实持仓，其次服务少量观察名单。

每日输出只保留三个核心目标：

- 组合主结论
  - 当前市场是否适合继续进攻
  - 当前总仓位是否合理
  - 哪些持仓应该继续拿，哪些应该处理
- 观察池机会
  - 只在白名单中寻找 0-3 个值得试仓的中期机会
  - 明确说明为什么是现在，不是之前或之后
- 风险撤退机制
  - 只在中期结构破坏或系统性环境恶化时，明确发出收缩提示

## Architecture

系统拆为 5 层，研究、决策、验证、报告彻底分离。

### 1. Account And Universe Layer

统一账户配置输入，拆分为：

- `portfolio`: 真实持仓
- `watchlist`: 少量观察名单
- `portfolio_state`: 现金、整数手、账户约束
- `strategy.swing`: 风格、最大持仓数、试仓额度、风险缓冲

这一层只负责账户事实，不做策略判断。

### 2. Research Scoring Layer

对持仓和观察池做统一的标准化打分，输出结构化证据，不直接给买卖动作。

评分维度分为：

- 市场状态：`进攻 / 均衡 / 防守 / 撤退`
- 标的质量：趋势、相对强度、回撤质量、量价结构、消息催化
- 角色定位：`核心 / 卫星 / 观察`

该层的输出必须是可追踪字段，而不是自然语言黑箱。

### 3. Decision And Execution Layer

这是主引擎。

它接收研究评分和账户约束，翻译成真实可执行动作：

- 对持仓：
  - `继续持有`
  - `逢回调加仓`
  - `减仓锁盈`
  - `退出观察`
- 对观察池：
  - `继续观察`
  - `进入试仓区`
  - `转正式仓`

任何动作必须同时经过三层约束：

- 市场层约束
- 标的层约束
- 账户层约束

这样系统不会出现“标的强，但仓位已满”“市场撤退，却还建议追击”的失真行为。

### 4. Validation Layer

验证层单独存在，不和日报主流程耦合。

它包含三类输出：

- 事件研究
  - 沿用现有 `10/20/40` 交易日窗口统计
  - 验证某类动作的统计优势
- 组合回测
  - 模拟 `T` 日收盘发信号、`T+1` 开盘执行
  - 模拟 A 股 `T+1`、整数手、手续费、滑点、现金路径
- Walk-forward
  - 使用滚动窗口验证规则稳定性
  - 防止只在某一段行情中看起来有效

验证结果必须反哺主引擎，例如：

- 某类 setup 历史表现差，则降级推荐
- 某类 setup 在某种市场状态里显著有效，则允许更积极

### 5. Report And Delivery Layer

日报只负责把结构化结论翻译成易执行文本。

LLM 只做解释层，不再决定核心动作。

中长期主报告固定为 5 个部分：

- 今日结论
- 账户动作
- 持仓处理
- 观察池机会
- 风险清单

## Mode Strategy

`swing` 成为唯一的主投资模式。

其他模式重新定位：

- `morning`: 仅做背景信息输入，不做主决策
- `midday`: 仅做盘中观察，不给中长期主动作
- `preclose`: 做次日执行准备
- `close`: 作为次日 `swing` 的主要数据基础

换言之，真正的投资动作建议主要在收盘后形成，并用于下一交易日执行。

## Data Model Changes

### Portfolio Schema

现有 `portfolio` 继续保留，用于真实持仓。

新增 `watchlist`：

```yaml
watchlist:
  - code: "512660"
    name: "军工ETF"
    strategy: "trend"
    priority: "high"
```

### Strategy Schema

在 `strategy.swing` 下新增更贴合中长期投资的控制项：

- `max_total_exposure`
- `min_cash_buffer`
- `max_single_position`
- `max_watchlist_adds_per_day`
- `max_satellite_positions`
- `candidate_limit`

这些字段用于将“激进”落到明确的账户边界。

## Decision Rules

### Holdings

持仓只允许四种主结论：

- `继续持有`
- `逢回调加仓`
- `减仓锁盈`
- `退出观察`

### Watchlist

观察池只允许三种主结论：

- `继续观察`
- `进入试仓区`
- `转正式仓`

### Rule Hierarchy

决策按优先级执行：

1. 市场是否允许进攻
2. 标的中期结构是否健康
3. 当前账户是否还有仓位和现金空间
4. 历史验证是否支持当前动作

## Validation Design

### Event Study

继续使用当前事件研究方法，但将动作类型与市场状态绑定统计：

- `逢回调加仓` 在 `进攻/均衡/防守` 下的收益与回撤
- `减仓锁盈` 后续是否显著减少回撤
- `进入试仓区` 是否具有正向超额收益

### Formal Backtest

引入正式回测模块，优先使用 `Backtrader` 风格约束建模：

- `T` 收盘决策
- `T+1` 开盘成交
- 手续费、印花税、滑点
- 整数手约束
- 现金与多标的组合路径

### Walk-forward

按时间切片滚动训练和验证：

- 训练窗：估计策略阈值和动作质量
- 测试窗：验证未来样本
- 多轮滚动后输出稳定性统计

## Reporting Design

### New Swing Layout

主报告结构固定如下：

1. `今日结论`
   - 用一句话总结现在应继续进攻、边打边看，还是收缩防守
2. `账户动作`
   - 建议总仓位区间
   - 是否允许新增试仓
   - 今天或明天优先处理的 1-2 个动作
3. `持仓处理`
   - 当前结论
   - 原因
   - 建议动作
   - 关键风险线
   - 触发后怎么办
4. `观察池机会`
   - 最多 3 个
   - 为什么现在进入试仓区
   - 适合试多少
   - 什么条件失效
5. `风险清单`
   - 最重要的 1-3 个系统性风险点

## Error Handling

- 数据不足时，不伪造观察池机会
- 回测样本不足时，清晰标注 `验证样本不足`
- 观察池为空时，明确展示 `当前无新增候选`
- LLM 失败时，结构化建议和回测结论仍可独立输出

## Testing Strategy

实现必须采用 TDD，并覆盖：

- 配置加载：`watchlist` 与新策略字段
- 决策层：持仓动作与观察池动作
- 回测层：`T+1`、滑点、手续费、整数手、现金路径
- 报告层：新 `swing` 结构和简化语言
- 回放：`--mode swing --replay --dry-run`

## Success Criteria

重构完成后，系统应满足：

- 以持仓和少量观察池为中心输出中长期动作
- 输出比当前版本更少术语、更强执行性
- 规则动作可被历史事件研究和正式回测验证
- 验证结果能反向影响动作建议
- 保留现有自动推送和 GitHub Actions 主链路
